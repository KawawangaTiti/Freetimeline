## Section 0 — How this audit was done

This audit was performed by **13 independent specialist reviewers**, each assigned a slice of the site, run in parallel and then reconciled into one plan. Critically:

- **The two engines were read in full, not sampled** — `js/universe-timeline.js` (9,981 lines) and `js/biography-timeline.js` (9,141 lines) were each read end-to-end by a dedicated reviewer.
- **Each page was reviewed individually** — `universe.html` (6,234 lines) and `biography.html` (6,383 lines) each had a dedicated reviewer; `index`, `about`, `contact`, `pricing`, `roadmap`, `guides`, `terms`, `privacy`, `cookies` and the 5 guide articles were each read on their own. See **Section 2B** for the per-page results.
- **The owner's two named complaints were root-caused specifically** — the Biography "stray lines across the middle" and the Universe "oversized bottom bar that doesn't work" (findings BS-1, BE-9, UE-2, US-1/US-2, MOB-3).

`PARITY_AUDIT.md`, `XSS_AUDIT.md`, `LIGHTHOUSE_AUDIT.md`, `A11Y_AUDIT.md`, `I18N_PLAN.md` — none exist. `package.json`, `vercel.json`, `.github/workflows/`, `CNAME`, `404.html`, `_headers`, `_redirects`, `netlify.toml` — **none exist.** The site has **no build step, no CI, and no committed deploy configuration of any kind** (see dimension **K**).

### How to read a finding

Every finding has an **ID** (e.g. `UE-1` = Universe-Engine #1, `BS-1` = Biography-Shell #1, `MOB-1` = Mobile #1, `SEC-01` = Security #1, `LEG-01` = Legal #1, `DOC-001` = parity/Docs #1). Where a finding matches an issue already logged in `PROJECT_REVIEW.md`, the existing **#nnn** is shown in the *Ref* column so you don't fix the same thing twice.

**Severity:** 🔴 Critical (breaks/loses data or law) · 🟠 High (clearly broken or risky) · 🟡 Medium (visible/should fix) · 🟢 Low (polish) · ⚪ Info (note).  
**Effort:** XS (minutes) · S (≈1 hr) · M (a session) · L (multiple sessions) · XL (a project of its own).

10-14 calendar weeks to a fully-fixed, ship-quality version.

Reasoning. The schedule is dominated NOT by raw coding hours (Claude Code writes the code fast) but by (1) the owner manually verifying each phase on a real phone and a real desktop before moving on, (2) the deliberate one-commit-per-issue cadence, and (3) the two structural traps that cannot be rushed: consolidating the five/four mobile UI systems (Phase 2) and the lossy-import / dual-undo data-integrity work (Phase 4), both of which need careful manual regression testing of saved data. Phases 1-3 (visual emergency, mobile foundation, accessibility) are ~4-5 weeks. Phase 4 engine/data-integrity is the riskiest single block at ~2-3 weeks because every fix touches saved localStorage and must be tested with real backups. Phases 5-9 (parity, performance, security headers, deploy, polish/QA) are ~4-5 weeks.

Lower bound 10 weeks assumes the owner is disciplined, hits no surprises, and the mobile consolidation goes cleanly. Upper bound 14 weeks is realistic and the number to plan around: it leaves slack for the owner-testing bug loop, for a deploy-host decision (GitHub Pages vs Vercel changes Phase 8), and for the inevitable real-device discoveries. Do NOT promise under 10 weeks; the verification overhead for a non-technical owner is the binding constraint, not the code.

### Dimension A — Visual / UI bugs

*Score 3.5/10 · 14 findings: 🔴 0 · 🟠 5 · 🟡 6 · 🟢 3 · ⚪ 0.* Both apps ship literal visible defects: Biography paints a full-width 'stray line' across a blank timeline (BS-1) and bright year-bands/dust that survive Reading Mode (BE-9), the Universe minimap is an always-on 100px bar nobody asked for (UE-2), the named 'Inter' font is never loaded (US-7), and #kbd-hint/scrollbar strips read as more stray marks.

| ID | Sev | Eff | Location | Issue → Root cause → Fix | Ref |
|---|:--:|:--:|---|---|:--:|
| `BS-1` | 🟠 | S | js/biography-timeline.js:1731-1733; js/biography-timeline.js:1189-1190; js/biography-timeline.js:5017 | **OWNER COMPLAINT ROOT CAUSE — 'stray lines across the middle of the screen'. On a fresh visit / blank template the timeline has exactly ONE life track ('Untitled', created at js line 5017). drawTracks() draws a full-viewport-width 1px horizontal 'Track bottom separator' for every visible track at the band BOTTOM: g.moveTo(0, bot); g.lineTo(c.width, bot). With a single centred track, bot sits near the vertical centre of the canvas, so the user sees a 1px horizontal line stretching edge-to-edge across the middle of the screen, plus the ruler base line at RULER_H (lines 1189-1190). ## Section 3 — Cross-cutting themes

*These group many findings under a single root cause — fix the cause once and dozens of symptoms disappear.*

## Cross-cutting themes (grouped by single root cause)

### Theme 1 — Too many competing mobile UI systems layered by successive ports
**Root cause:** each port/sprint bolted on a NEW mobile shell instead of replacing the old one. Universe has FIVE systems (#mob-header/#mob-bar/#mob-drawer, compact #tlb-sheet, #bot-bar, pill-bar/feed, FT_MOBILE_PASS); Biography has FOUR. A later `display:none !important` then silently kills the older system that the visible buttons still call.
**Findings:** US-1, US-2, US-3, US-4, US-9, US-10, US-11, US-14, BS-2, BS-3, BS-4, BS-12, BS-20, MOB-3, MOB-5, MOB-11, MOB-14, DOC-005, DOC-010, DOC-013.
**Consolidated fix:** pick ONE mobile system per app (Universe System B #tlb-sheet + pill-bar; Biography its compact header), delete `#bot-bar` and the dead System-A markup/CSS, unify all breakpoints to a single value, then re-test every mobile button. **Total effort: L (the single highest-leverage change on the site).**

### Theme 2 — Missing viewport/safe-area/DPR foundation makes mobile blurry and overflowing
**Root cause:** the head never sets `viewport-fit=cover`, the canvas never multiplies by devicePixelRatio, and layout uses static `100vh` instead of `100dvh`.
**Findings:** MOB-1, MOB-4, MOB-7, MOB-10, MOB-13, MOB-15.
**Consolidated fix:** add `viewport-fit=cover` to every viewport meta, scale the canvas backing store by DPR with a ctx transform, switch `100vh`->`100dvh` (with a vh fallback), enforce 44px min touch targets. **Total effort: M.**

### Theme 3 — render() monkey-patch chains + always-on rAF loops cause stutter and stray marks
**Root cause:** every feature wrapped the global render binding instead of registering on a shared scheduler, and ambient/pill/feed modules each started their own permanent requestAnimationFrame loop with no dirty-flag or visibility guard.
**Findings:** UE-2, UE-9, UE-13, UE-14, BE-4, BE-9, BE-14, MOB-2, MOB-16, US-14, BS-15.
**Consolidated fix:** introduce one dirty-flag render scheduler; convert the wrappers into subscribers; gate ambient/feed loops on movement + document.visibilityState; patch the band/dust overlay contexts for Reading Mode (kills the stray-marks bug too). **Total effort: M-L.**

### Theme 4 — Defence-in-depth security gap: no CSP + trusted localStorage + a few missed escapes
**Root cause:** the import-FILE path was hardened (#052/#053) but the persistence boundary, a handful of person fields, and the response-header layer were not — and there is no CSP backstop.
**Findings:** SEC-01, SEC-02, SEC-03, SEC-04, SEC-06, SEC-07, SEC-08, UE-6, UE-16, BE-3, BE-16, DEP-3.
**Consolidated fix:** add a strict meta CSP (and a `_headers`/`vercel.json` header when host is known), run `ftValidateImport` on `Store.load()` too, wrap ch.photo/ch.color in esc(), escape `</script>`/markers in saveHTML, fix the `[<>" -]` regex range, drop svg+xml from the URL whitelist, strip the baked-in Cloudflare scripts. **Total effort: M.**

### Theme 5 — Two 95%-duplicated engines cause every parity bug and double every fix
**Root cause:** Biography was forked from Universe and edited in parallel with no shared module; renames drifted (tone->emotionalTone, connections vs map, Organizations vs Groups) and copied stats code kept the old field/category names.
**Findings:** UE-7, BE-1, BE-2, DOC-001, DOC-002, DOC-003, DOC-004, DOC-006, DOC-008, DOC-012, DOC-013, MOB-16, #068.
**Consolidated fix (near-term):** do NOT merge the files yet; instead fix each drift bug in BOTH files in the same commit and add a one-line comment marking the shared block. <a id="section-7--what-not-to-do"></a>
## Section 7 — What NOT to do

## What NOT to do

These are the moves most likely to break this specific project. Show this list to anyone (including a future Claude session) before they touch the code.

### Do NOT break saved localStorage data
- Do NOT rename or restructure the localStorage keys (`inf_universe_v4`, `inf_biography_v1`, `ft_uni_view_v1`, `ft_bio_settings_v1`, etc.) without a migration path. The owner's real data lives there; a rename = silent total data loss on next load.
- Do NOT change the shape of the saved state object (rename `universes`->`lifeTracks`, drop fields) without keeping backward-compatible readers. Biography already reads both `lifeTracks` and `universes` for exactly this reason — preserve that tolerance.
- Do NOT 'clean up' the import validator by tightening the whitelist; the bug (UE-4) is that it drops TOO MANY fields, not too few. Only ADD fields to the whitelist.
- Do NOT make Reading Mode, undo, or import changes without a fresh JSON backup loaded and tested first.

### Do NOT violate the no-build, simple-engine architecture
- Do NOT introduce a build step, bundler, framework (React/Vue/Svelte), or npm dependency for the app. There is no package.json by design; it deploys as static files anywhere. Adding a toolchain makes it un-deployable for a non-technical owner and breaks the 'open the file and it works' property (the save-as-HTML feature literally depends on this).
- Do NOT convert the engines to ES modules casually. The render() monkey-patch chain currently works ONLY because top-level function declarations are also window properties in a classic script (UE-13). Module-ifying without first removing those patches will silently break minimap/hash sync.
- Do NOT add a server, database, or backend 'to fix' anything. The entire privacy proposition and the legal copy depend on data never leaving the browser. A backend would invalidate the policies and the brand.
- Do NOT pull in a CSS framework or icon-font dependency to fix visuals. The pages already ship ~3,800 lines of inline CSS; adding more dependencies makes the dead-CSS problem worse and complicates the static deploy.

### Do NOT complicate the static deploy
- Do NOT adopt a host that requires a build/CI pipeline the owner can't operate. GitHub Pages (static) or Vercel (zero-config static) are the right tier. If GitHub Pages is chosen, accept that custom response headers are impossible there and rely on the meta CSP — do not add a reverse proxy just for headers.
- Do NOT hardcode absolute `https://freetimeline.pt/...` asset paths; keep root-relative so preview deployments work.
- Do NOT deploy the whole repo branch as-is — the private `.md` docs must be excluded.

### Do NOT split the monoliths yet
- Do NOT merge `universe-timeline.js` and `biography-timeline.js` into one shared module DURING Phases 1-8. They are 95% duplicated (#068) and merging them is an XL change that would touch every bug you are simultaneously trying to fix, making regressions impossible to attribute. Fix bugs in BOTH files in the same commit instead.
- Split them ONLY after Phase 9 ships and is stable, as its own dedicated project (see the maintenance plan), with full backups and a test of every feature in both apps afterwards. The merge is a maintainability win, not a user-facing fix, so it must never block shipping.
- Likewise do NOT do the inline-onclick -> addEventListener refactor (#069) as part of a feature phase; it is large, it interacts with the script-load-order fragility, and a half-done conversion leaves dead buttons. Schedule it as its own task after launch, paired with the CSP work.

### Do NOT 'improve' things the audit confirmed are already correct
- The AdSense consent gating, ads.txt, the no-placeholder-slots state, and the import-FILE sanitiser are SOUND. Do not refactor them. Re-touching them risks reintroducing the very issues (#001/#004) that were already closed.
- Do not re-file the already-closed parity features (onboarding #043, Reading Mode desktop #019, Continuity Tour #020, shared confirm #018) as new gaps — they are present in both apps.

## Section 8 — Maintenance plan

## Ongoing maintenance plan (after all phases ship)

### Day-to-day: user-reported bugs
- Keep using the `🐛 Bugs found during owner testing` section at the bottom of POST_LAUNCH.md. For each real-use problem, record: what you did, the device/browser, and what you expected vs what happened. A screenshot helps enormously.
- Fix loop: open a fresh Claude Code session, say `Read POST_LAUNCH.md and the latest 🐛 bug — diagnose and fix, harsh mode, one commit, show me the diff`. Always export a JSON backup from the affected app first.
- After any fix that touches data or the canvas, re-run the two-minute smoke test: load a backup, make an edit, undo it, export and re-import, and check the change survived.

### Routine upkeep (every ~3 months)
- Keep policy and code in sync (the #012 standing note): the moment you add ANY analytics or change ad behaviour, update privacy.html and cookies.html the same day. The biggest legal risk on this site is the policy describing something the code does not do (LEG-01 class).
- Re-reconcile the cookie table with the actual localStorage keys whenever you add a feature that stores anything new (grep the codebase for `localStorage.setItem` and compare to the table).
- Refresh sitemap.xml / feed.xml lastmod dates when you publish content; these are hand-maintained.
- Re-test the consent banner + ad gating after any AdSense console change.

### When to re-audit
- Run a fresh full audit (a) before any major new feature ships, (b) after the eventual engine-merge or onclick refactor, (c) once a year regardless, and (d) immediately if AdSense flags the site or a user reports data loss.
- A lightweight self-audit after every batch of fixes: open every page with the browser console visible and confirm zero red errors and zero CSP violations; run Lighthouse on the homepage, one app, and one guide.

### When to split the monoliths (#068) / do the onclick refactor (#069)
- Only after the site has been live and stable for a few weeks post-Phase-9, with no open data-integrity bugs.
- Trigger to actually do it: when you notice you are fixing the SAME bug twice (once per engine) more than a couple of times, the duplication is costing more than the merge would. That is the signal #068 is now worth paying down.
- Do it as its own dedicated, backed-up project — extract the shared 95% into one file that both apps import, keeping the per-app differences (palette, vocabulary, the Universe-only switcher) as small config. Pair it with the #069 inline-handler -> addEventListener conversion, because both are prerequisites for a strict, tighter CSP. Afterwards, manually exercise EVERY feature in BOTH apps.

### When a rewrite is warranted (and when it is NOT)
- A full rewrite is NOT warranted today. The engine is feature-rich, defensively coded at the edges, and its defects are concrete and fixable; throwing it away would discard working AdSense/consent/import infrastructure and risk the owner's saved data.
- Consider a rewrite ONLY if ALL of these become true at once: (1) the two engines have drifted so far that fixes no longer port cleanly even after the merge, (2) you want a fundamentally different rendering approach (e.g. SVG/DOM events instead of canvas) for accessibility that the current canvas cannot deliver, and (3) you have the time for a multi-month project with a full data-migration plan. Until then, prefer incremental fixes.
- If a rewrite ever happens, the non-negotiable carry-overs are: stay no-build/static, keep the exact localStorage key shapes (or ship a migration), preserve export/import round-trip fidelity, and keep the privacy-by-architecture (no backend) promise that the entire brand and legal posture rest on.

### Keep the safety habits
- One commit per issue, descriptive `Fix #xxx:` messages — this is what makes `git revert` a safe undo button for a non-technical owner.
- Never push straight to production; use a preview/staging deploy and only promote after a quick phone+desktop check.
- Keep an off-machine JSON backup of your own real timelines; the app is localStorage-only, so a cleared browser or a dead laptop loses everything otherwise.

<a id="section-9--glossary"></a>
## Section 9 — Glossary (plain language)

| Term | Plain-language meaning |
|---|---|
| **localStorage** | A small storage box inside each visitor's own browser where the app saves their timeline. Nothing leaves their computer. If a 'key' (the box's label) is renamed, the saved work appears lost. |
| **Key (storage key)** | The label on a localStorage box, e.g. `ft_universe_state`. Renaming one orphans existing saved work. |
| **innerHTML** | A way the code injects HTML into the page. If user text is injected without cleaning, it can become a security hole (see XSS). |
| **esc() / escaping** | Cleaning user text so characters like `&lt;` can't turn into live code. The fix for most XSS holes. |
| **XSS (cross-site scripting)** | A bug where attacker text becomes running code in someone's browser — can steal data or hijack the page. |
| **Sanitiser / whitelist** | Code that inspects imported data and keeps only known-safe fields/values. `js/ft-import-validate.js` is this site's sanitiser. A whitelist that forgets a field silently deletes that field on import. |
| **CSP (Content-Security-Policy)** | A safety net, set in the page head or server headers, that tells the browser which scripts/images are allowed. With it, a single missed escape is far less dangerous. This site currently has none. |
| **Round-trip** | Export your data to a file, then import it back. A *lossless* round-trip returns exactly what you had. This site's is currently lossy (loses some fields). |
| **Undo/redo stack** | The list the app keeps so Undo can step back. Universe has two competing lists that disagree — the cause of silent corruption (UE-1). |
| **Canvas** | The drawing area where the timeline is painted pixel-by-pixel (not normal web elements). Needs special handling for sharpness and touch. |
| **DPR / devicePixelRatio** | How many real pixels a phone packs per CSS pixel (2–3× on modern phones). If the canvas ignores it, the timeline looks blurry on mobile. |
| **Viewport** | The visible area of the page on a device. The `&lt;meta name=viewport&gt;` tag controls how the page fits the screen. |
| **viewport-fit=cover** | A viewport setting required for iPhone notch handling to work. Without it, all 102 'safe-area' rules in this site do nothing (MOB-1). |
| **safe-area-inset / env()** | Padding values iOS provides so content avoids the notch and home bar. Only active when `viewport-fit=cover` is set. |
| **100vh vs 100dvh/100svh** | `100vh` is a fixed full-screen height that mis-sizes mobile browsers (toolbar overlaps). `100dvh/100svh` adapt to the actual visible height. |
| **Breakpoint** | The screen width (e.g. 768px) where the layout switches to its mobile version. This site uses several inconsistent values (720/767/820). |
| **clamp()** | A CSS function that scales a value between a min and max — handy for fonts, but a bad clamp can make text break on phones. |
| **rAF / requestAnimationFrame** | A loop that redraws ~60×/second. Several here run permanently even when idle, draining battery (G dimension). |
| **Monkey-patch** | Re-wrapping an existing function to add behaviour. Done 3–4 times to `render()` here, which multiplies the work per frame. |
| **DOM** | The live tree of elements that makes up the page. Buttons 'bind' to handlers in the DOM. |
| **Handler / event listener** | The code that runs when you click/tap a control. A 'dead' control is one whose handler is missing or hidden (D dimension). |
| **IIFE** | A block of script that runs once immediately. Two of them created the duplicate consent banner and the duplicate undo system. |
| **ARIA** | Extra labels in the HTML that let screen readers announce controls to blind users (C dimension). |
| **Focus indicator / focus-visible** | The visible outline showing which control the keyboard is on. Required for accessibility; this site has a good baseline. |
| **Skip link** | A hidden 'skip to content' link for keyboard users. Present here but duplicated in places. |
| **Contrast ratio** | How readable text is against its background. AA standard wants ≥4.5:1 for body text. Some labels here are ~2.2:1 (too faint). |
| **z-index** | Which element stacks on top of which. Wrong values cause overlaps (A dimension). |
| **Canonical tag** | A `&lt;link rel=canonical&gt;` telling Google the official URL for a page, preventing duplicate-content confusion. Missing on the two apps. |
| **OpenGraph / Twitter card** | Tags that control the preview image/title when a link is shared on social media. |
| **Sitemap / RSS feed** | `sitemap.xml` lists your pages for search engines; `feed.xml` is a subscribe-able list of updates. Both are hand-maintained here and drift. |
| **404 page** | The page shown for a broken/unknown link. This site has none, so visitors hit a generic host error (K dimension). |
| **_headers / _redirects** | Files some hosts (Cloudflare/Netlify) read to add security headers or URL redirects. None exist here yet. |
| **GitHub Pages / Vercel / Cloudflare Pages** | Services that publish your static files to the web. The plan recommends choosing ONE. |
| **GDPR** | EU/Portugal privacy law. Requires real consent (with a reject option) before advertising/tracking cookies load (J dimension). |
| **AdSense slot** | A specific ad placement ID from Google. Placeholder IDs (all zeros) serve no ads. |
| **Git / commit / branch / revert** | Git is the version history. A *commit* is one saved change; a *branch* is a safe parallel copy; *revert* undoes a commit cleanly. Your safety net. |
| **Monolith** | A single very large code file. The two engines (~9–10k lines each) are monoliths. |
| **Regression** | A new bug accidentally introduced while fixing something else. Verify after each phase to catch these. |
| **Parity** | Two sibling things behaving the same. Universe and Biography should match except where the data demands difference (E dimension). |
